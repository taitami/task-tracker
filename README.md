# Лабораторна робота №1: Розгортання Web-сервісу з автоматизацією

## 1. Варіант індивідуального завдання
Розрахунки для номера N = 16:
- **V2** = (16 % 2) + 1 = 1 (Спосіб конфігурації: Аргументи командного рядка; База даних: MariaDB)
- **V3** = (16 % 3) + 1 = 2 (Тематика: Task Tracker)
- **V5** = (16 % 5) + 1 = 2 (Порт застосунку: 5200)

## 2. Призначення застосунку
Сервіс `mywebapp` є простим Task Tracker'ом, який дозволяє відстежувати задачі. Написаний на TypeScript з використанням Express.js та MariaDB. Підтримує Content Negotiation (віддає `text/html` або `application/json` залежно від заголовка `Accept`).

## 3. Документація по API
- `GET /` - Кореневий ендпоінт. Повертає список доступних API (тільки `text/html`).
- `GET /tasks` - Повертає список усіх задач (id, title, status, created_at).
- `POST /tasks` - Створює нову задачу. Очікує параметр `title`.
- `POST /tasks/<id>/done` - Змінює статус задачі на `done`.
- `GET /health/alive` - Перевірка життєздатності (завжди 200 OK). Заблоковано ззовні.
- `GET /health/ready` - Перевірка підключення до БД (200 OK або 500 Error). Заблоковано ззовні.

## 4. Документація по розгортанню
- **Базовий образ ВМ**: Офіційний образ Ubuntu 22.04 LTS (Server).
- **Вимоги до ресурсів**: 1 vCPU, 1 GB RAM, 10 GB Disk.
- **Вхід на ВМ**: За замовчуванням по SSH за допомогою ключа, користувач `ubuntu`. Після виконання автоматизації вхід можливий користувачами `student`, `teacher`, `operator`.
- **Інструкція з розгортання**:
  1. Підключитися до віртуальної машини.
  2. Склонувати цей репозиторій у `/opt/mywebapp` (або іншу директорію).
  3. Перейти в директорію `mywebapp` та виконати компіляцію:
     ```bash
     cd /opt/mywebapp/mywebapp
     npm install
     npm run build
     ```
  4. Повернутися в корінь репозиторію і запустити скрипт автоматизації від імені суперкористувача:
     ```bash
     cd /opt/mywebapp
     sudo bash deploy.sh
     ```

## 5. Тестування розгорнутої системи
Після успішного розгортання можна перевірити коректність роботи сервісу, виконуючи запити до IP-адреси віртуальної машини (на порт 80, оскільки Nginx проксіює запити):
- Перевірка отримання списку задач у форматі JSON:
  `curl -H "Accept: application/json" http://<IP_ADDRESS>/tasks`
- Перевірка створення нової задачі:
  `curl -X POST -d "title=Complete Lab 1" http://<IP_ADDRESS>/tasks`
- Перевірка обмеження доступу (має повернути 403 Forbidden):
  `curl -I http://<IP_ADDRESS>/health/alive`

## Лабораторна робота №3: Налаштування CI/CD

У цій лабораторній роботі налаштовано процеси безперервної інтеграції (CI) та розгортання (CD) для сервісу Task Tracker.

### 1. Continuous Integration (CI)
Пайплайн автоматично запускається на кожен коміт та Pull Request у гілку `main` і виконує наступні кроки:
- **Лінтери:** Перевірка коду TypeScript (ESLint), Dockerfile (Hadolint) та bash-скриптів (Shellcheck).
- **Тестування:** Запуск автоматичних тестів за допомогою Vitest та Supertest.
- **Покриття коду:** Перевірка покриття коду тестами (мінімум 40%). Звіт завантажується як артефакт GitHub Actions.
- **Збірка та Публікація:** При успішному проходженні перевірок збирається Docker-образ та публікується у GitHub Container Registry (GHCR) з відповідними тегами (`latest`, `sha-<hash>`, або `stable`, `<tag>`).

### 2. Налаштування Self-hosted Runner
[cite_start]Для розгортання застосунку використовується окрема віртуальна машина з Ubuntu 24.04, яка виступає в ролі GitHub Self-hosted Runner[cite: 231]. 

**Інструкція з налаштування:**
1. Створити нову віртуальну машину (Ubuntu 24.04).
2. [cite_start]Завантажити скрипт налаштування: `wget https://raw.githubusercontent.com/<ТВІЙ_ЛОГІН>/mywebapp/main/scripts/setup_runner.sh` (якщо ти робив скрипт)[cite: 235].
3. Виконати скрипт: `sudo bash setup_runner.sh`.
4. [cite_start]Вручну додати Runner через налаштування репозиторію GitHub (Settings -> Actions -> Runners)[cite: 236].
5. Додати на Runner приватний SSH-ключ для доступу до Target Node.

### 3. Continuous Delivery (CD)
Пайплайн CD запускається виключно на створення анотованих тегів (наприклад, `v1.0.0`) і виконується на Self-hosted Runner.
- Runner підключається до Target Node (ВМ із застосунком) по SSH.
- Виконується завантаження нового `stable` образу з GHCR.
- Оновлюється та перезапускається `systemd-unit` для Docker-контейнера.
- Запускається скрипт `verify.sh`, який перевіряє доступність сервісу (HTTP 200) та роботу обмежень Nginx (HTTP 403 на `/health`).

## Лабораторна робота №4: IaC (Terraform) та Configuration Management (Ansible)

Інфраструктура проєкту розгортається на двох віртуальних машинах (`worker` та `db`), створених автоматично.

### Запуск Terraform
1. Перейдіть до папки `terraform`: `cd terraform`
2. Ініціалізуйте провайдер: `terraform init`
3. Створіть ВМ: `terraform apply` (підтвердіть введенням `yes`).
4. Скопіюйте IP-адреси ВМ, які з'являться на екрані.

### Запуск Ansible
1. Перейдіть до папки `ansible`: `cd ansible`
2. Відкрийте файл `inventory.ini` та впишіть актуальні IP-адреси, які ви отримали від Terraform.
3. Запустіть налаштування командою: 
   `ansible-playbook -i inventory.ini playbook.yml`

*Усі налаштування застосовуються декларативно (використовуються модулі apt, user, systemd, template). База даних не доступна ззовні, а застосунок динамічно отримує її IP-адресу з Ansible templates.*
